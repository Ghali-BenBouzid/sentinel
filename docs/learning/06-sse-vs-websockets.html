01 / starting point

The question this started with

web/src/api/sse.ts reads a stream's raw bytes with response.body.getReader() and hand-splits frames on \n\n. The browser ships a built-in class for exactly this, EventSource. Why isn't Sentinel using it?

Short answer, unpacked below: EventSource can only send GET requests with no body and no custom headers. Sentinel's first message of a session goes out as POST /sessions with a JSON body ({"message": ..., "autonomy": ...}) and the SSE stream is that POST's response. There's no way to express that with EventSource at all — so the hand-rolled parser isn't a missed shortcut, it's the only option once the transport needs a request body.

02 / protocol

What SSE actually is

Server-Sent Events is not a separate protocol the way WebSockets is — it's a text convention layered on top of a perfectly ordinary HTTP response. The server sets Content-Type: text/event-stream, never closes the response body, and writes frames to it over time. Each frame is one or more field: value lines followed by a blank line:

data: {"event": "message", "data": {"text": "..."}} ← the payload; SSE itself doesn't care what's inside (blank line terminates the frame) event: update ← optional: names the event, lets EventSource dispatch by type id: 42 ← optional: client echoes this back on reconnect retry: 3000 ← optional: reconnection delay in ms data: {"event": "done", "data": {}}

Sentinel only ever uses the data: field. Its _sse() helper in sentinel/api/app.py is the entire encoder: 

def _sse(event: str, data) -> str:
    return f"data: {json.dumps({'event': event, 'data': data})}\n\n"

The event's type lives inside the JSON payload ({"event": "message", ...}), not in SSE's own event: field. That's a real design choice, not an oversight — see §7.

Captured straight from a real running session this session, over plain curl:

$ curl -sN -X POST http://127.0.0.1:8000/sessions \
    -H "content-type: application/json" -d '{"message":"hello there"}'

data: {"event": "message", "data": {"text": "To begin, I need to gather the run configuration..."}}

data: {"event": "done", "data": {}}

That's the whole protocol on the wire: an ordinary HTTP request, an ordinary streaming HTTP response, readable with nothing more exotic than curl -N (the flag just turns off curl's own output buffering). No handshake, no upgrade, no separate port.

03 / browser api

Why not EventSource?

EventSource is a real, useful browser API — it gives you automatic reconnection with backoff, and dispatches typed event: frames as native DOM events, for free. It has three hard restrictions, though, and Sentinel's transport violates all three:

  • GET only. No way to issue a POST.
  • No request body. Everything would have to go in the URL — awkward and length-limited for a chat message, worse for the JSON config POST /sessions/{id}/autonomy sends.
  • No custom headers. Not a blocker for Sentinel today (no auth header yet), but would be the moment any auth gets added.

This is common enough that a small ecosystem exists just to work around it — Microsoft and Azure both publish a fetch-event-source library that layers POST + headers + controllable retry on top of fetch(), parsing frames by hand internally, exactly like web/src/api/sse.ts does. Sentinel's version is about 30 lines because it only needs to support one shape of stream; reaching for a dependency here would have bought back-pressure handling and reconnection Sentinel doesn't currently use, at the cost of one more thing to vendor.

what Sentinel gives up by not using EventSource

Automatic reconnection. If a stream drops mid-turn today, consumeStream's catch block turns it into an error event and the user has to resend — there's no retry-with-Last-Event-ID recovery. Worth knowing as a deliberate current limitation, not a bug: see §7.

04 / comparison

SSE vs WebSockets

The one-line version: SSE is server→client only, running over plain HTTP; WebSockets are a genuinely different protocol (ws:///wss://, negotiated via an HTTP upgrade handshake) giving full-duplex, bidirectional messaging over one persistent TCP connection.

DimensionSSEWebSockets
DirectionServer → client onlyFull duplex
TransportPlain HTTP, no upgradeHTTP upgrade to a distinct protocol
Client sends data viaA separate, ordinary HTTP requestThe same open socket
ReconnectionBuilt into EventSource, with Last-Event-ID replayNone built in — the app has to detect the drop and redial
Message framingText only, line-delimitedText or binary, app-defined framing
Firewall / proxy friendlinessHigh — it's just HTTPLower — some proxies/corporate networks block the upgrade
Debuggable withcurl -N, browser network tabNeeds a WS-aware client (wscat, devtools WS pane)

Does the "agents are moving to WebSockets" trend apply to Sentinel?

Current industry commentary notes that as AI products grow past basic chat into agent workflows with human-in-the-loop approval and multi-device use, teams increasingly reach for WebSockets. Sentinel already has human-in-the-loop (the confirm() / interrupt() rail) — so it's worth asking directly whether that applies here.

The trend is really about one specific gap: the server needing to push to a client that isn't mid-request — a background job finishing while the tab sits idle, or a second device needing to see a confirmation the first device raised, instantly. Sentinel doesn't have that shape today. Every SSE stream is the direct response body of a request the client just made (POST /sessions/{id}/message, /resume) — the confirmation card in ConfirmCard.tsx is answered by the same tab making another POST, not by an unprompted server push to an idle connection.

where this would actually flip

If run_monitor ever needs to notify an idle tab the moment it files an alert ticket, or two browser tabs need to see the same pending confirmation resolve live without a manual refresh, that's exactly the "push outside a request/response cycle" case WebSockets solve and SSE structurally can't. Worth remembering as the concrete trigger for revisiting this, rather than switching pre-emptively.

05 / the rest of the field

vs polling, HTTP/2 push, WebTransport

Polling and long polling

Plain polling (ask every N seconds) wastes requests and adds up to N seconds of latency on every update. Long polling (hold the request open until there's something to say, then respond and immediately re-open) gets closer to real push but re-pays a full HTTP request/response cycle for every single message. SSE keeps one request open and streams many messages down it — it's what long polling was approximating.

HTTP/2 server push — effectively dead

Worth naming because it sounds related and isn't: HTTP/2 defined a PUSH_PROMISE mechanism for a server to proactively send resources before the client asked. It's a different mechanism from SSE (browser-cache-level, not an application data stream), and it didn't work out — usage stayed under 1.3% of HTTP/2 sites, with mixed-to-negative real-world performance results. Chrome disabled it by default in 2022 and removed it; Firefox followed in October 2024. Not a factor in any current transport decision.

WebTransport — promising, not production-ready

Built on QUIC (HTTP/3's transport), WebTransport offers WebSocket-like bidirectional multiplexed streams without head-of-line blocking — genuinely the more modern design. Current assessment: server-side support across mainstream frameworks is still experimental going into 2026. Something to prototype with, not something to build Sentinel's transport on today.

06 / production concerns

Operational realities

The HTTP/1.1 six-connections-per-origin limit

Browsers cap concurrent connections to one origin at six under HTTP/1.1 (Chrome's default). An open SSE stream occupies one of those six for its entire lifetime, since the response never completes. Six open tabs each holding a live Sentinel session against the same origin would exhaust the pool and queue a seventh. HTTP/2 multiplexes many streams over one TCP connection and makes this limit irrelevant (Chrome defaults to ~100 concurrent streams per connection) — but Sentinel's dev server (uvicorn ... --port 8000, no TLS) speaks HTTP/1.1, so the limit is live today, just unlikely to be hit at one-user scale.

Reverse proxies buffer streams by default — and that breaks them

Not a concern yet (nothing sits in front of Sentinel's dev server today besides the browser and CORS), but the standard gotcha for whenever that changes: nginx buffers proxied responses by default, which silently turns a stream into "wait for the whole thing, then deliver it" — defeating the entire point. The fix is proxy_buffering off, an X-Accel-Buffering: no response header, proxy_http_version 1.1, and a long proxy_read_timeout so an idle-but-open stream isn't cut. Every proxy layer in the chain needs it, not just the first one.

Scaling past one process

SSE connections are long-lived, so a load balancer that distributes traffic randomly can strand a reconnecting client on a different server than the one holding its stream. The two production answers are sticky sessions (simple, but a server restart drops every client pinned to it) or a stateless design backed by a shared store like Redis pub/sub (more resilient, more moving parts). Not a decision Sentinel needs yet — it runs as one process today, and notably the SqliteSaver checkpointer this session's bug fixes revolved around is itself inherently single-writer, so multi-process would need to solve checkpoint storage before SSE fan-out becomes the bottleneck.

07 / left on the table

What SSE offers that Sentinel doesn't use

event: (typed events)

SSE can name each frame's type at the protocol level, letting EventSource dispatch listeners per type. Sentinel nests the type inside the JSON payload instead, parsed by one function.

sentinel/api/app.py: _sse()

id: + Last-Event-ID

Lets a reconnecting client tell the server exactly where it left off, so missed frames can be replayed. Requires EventSource (or reimplementing the reconnect logic) to matter at all.

web/src/api/sse.ts

retry: (backoff hint)

Server-controlled reconnect delay. Moot without automatic reconnection in the first place.

n/a today

None of this is a gap to close reflexively — it's the direct cost of the POST-with-a-body requirement in §3. The one worth revisiting first if it ever bites in practice is reconnection: today a dropped stream mid-turn just surfaces as an error bubble and the user resends by hand.

08 / quick reference

What to reach for

SituationReach for
Server streams a response to a request the client just madeSSE
Client needs to send data mid-stream, not via a new requestWebSockets
Server must push to a client that isn't mid-request (idle tab, other device)WebSockets
Infrequent updates, staleness of a few seconds is finePolling (simplest, often good enough)
Binary payloads (audio, video frames)WebSockets (SSE is text-only)
Must traverse strict corporate proxies/firewallsSSE (it's just HTTP)
Ultra-low-latency multiplexed streams, modern-only clients, willing to prototypeWebTransport (not production-ready yet)

Sentinel's actual traffic — one bounded stream per request, server-initiated within that request, no idle-client push yet — sits squarely in SSE's sweet spot. The open question isn't "should this be WebSockets," it's "does run_monitor or multi-tab sync ever need to push to a client that isn't already asking" — and that's a product decision, not a transport one, for whenever it actually comes up.
