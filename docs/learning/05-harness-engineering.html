01 / incident

The incident, decoded

Mid-conversation, the agent answered a user's question in prose and tried to call save_config() in the same turn — with no arguments. save_config requires four fields (framing, failure_threshold, reporting_cadence, success_metric) that the conversation hadn't gathered yet. Groq's API validates tool-call arguments against the tool's JSON schema on the server, before returning anything — so instead of a normal completion, it returned an HTTP 400 tool_use_failed. That exception propagated straight out of the model call, through create_agent's ReAct loop, and out the SSE endpoint as a raw exception dump.

Two things make this specific, not generic "LLMs sometimes mess up":

  • It is a provider-side rejection, not a LangChain parsing failure. A related but distinct LangChain gap exists too (GH #33504): when the model's JSON is malformed enough that LangChain's own parser can't build a tool_calls entry at all, it lands in invalid_tool_calls instead — and create_agent's routing only checks whether tool_calls is empty, so the turn silently ends without ever telling the model what went wrong. Different failure, same symptom: the model never sees its own mistake.
  • It is more likely on Groq specifically because of decoding mode. Groq's structured-output docs describe two modes: strict: true uses constrained decoding and guarantees schema-valid output; without it, the model free-generates JSON that is only validated after the fact, so malformed or incomplete tool calls slip through generation and get rejected at the gate. LangChain's default bind_tools() on ChatGroq does not turn strict mode on.

Why this matters more here than on a paid frontier model

Nothing in the current stack catches a model-call exception and turns it into something the model (or the user) can recover from. The confirmation rail (confirm() in tools.py) only guards successful tool calls from running without consent — it has nothing to say about a call that never reaches a tool at all.

02 / taxonomy

The failure map

Harness engineering research frames 2026 as the year this became its own discipline: designing the constraints, feedback loops, and verification around an agent so its mistakes get caught and corrected rather than reaching the user raw. Below is where Sentinel actually stands against each category — not a generic checklist.

Category What breaks Likelihood on free Groq Sentinel today Native fix
Model call failure Malformed/incomplete tool-call args rejected server-side; rate limits; timeouts likely shipped — ModelRetryMiddleware + ModelFallbackMiddleware + ModelFailureFormatterMiddleware ModelRetryMiddleware, ModelFallbackMiddleware
Invalid tool-call parsing JSON malformed enough that LangChain can't build tool_calls at all — lands in invalid_tool_calls, ignored by routing possible shipped — custom InvalidToolCallMiddleware custom middleware (no built-in fix yet)
Tool execution failure A tool raises, or receives schema-valid but semantically wrong arguments (bad model id, etc.) possible handled — tools return strings, never raise; ToolNode's own default catches anything that still escapes dropped — ToolRetryMiddleware can't see tool exceptions under create_agent's default ToolNode wiring, verified against installed source
Runaway loops / cost Agent keeps calling tools or re-prompting without converging rare shipped — ModelCallLimitMiddleware + ToolCallLimitMiddleware ModelCallLimitMiddleware, ToolCallLimitMiddleware
Context overflow Long conversations or big tool outputs exceed the context window rare today, grows with usage unbounded SummarizationMiddleware, ContextEditingMiddleware
Safety / consent Destructive or costly actions run without a human saying yes — shipped — HumanInTheLoopMiddleware, replacing the hand-rolled confirm() + interrupt() rail HumanInTheLoopMiddleware does the same thing built-in
Observability No way to see why the agent chose a tool call after the fact, or measure regressions — none — SSE events only, nothing persisted LangSmith tracing; DeepEval/Promptfoo for evals
Model staleness The pinned model gets deprecated or shut down upstream — shipped — swapped to openai/gpt-oss-120b / openai/gpt-oss-20b config-level model override (already exists), needs a swap

Concrete finding, not general advice

Groq's own deprecation page lists llama-3.3-70b-versatile — the model sentinel/llm/provider.py hardcodes as the "smart" tier for the groq provider — with a scheduled shutdown around 2026-08-16 on free/developer tiers, recommending openai/gpt-oss-120b or qwen/qwen3.6-27b as replacements. Worth confirming directly at console.groq.com/docs/deprecations before acting — but if accurate, that's about five weeks out from today. This is independent of the harness work below: it's a config change, not a reliability pattern.

Shipped. _MODELS["groq"] now points at openai/gpt-oss-120b / openai/gpt-oss-20b, confirmed with a live tool-call smoke test before anything else in the harness stack was built on top of it.

03 / langchain

LangChain's native toolkit

This is the direct payoff of having built on create_agent rather than a hand-rolled loop: every category above except "invalid tool-call parsing" already has a maintained, tested middleware class. middleware=[...] composes on create_agent exactly the way it composes today for the confirmation rail — these are additions, not a rewrite.

ModelRetryMiddleware

Retries a failed model call with exponential backoff. This is the direct fix for the save_config incident's failure mode — a transient rejection, not a broken tool.

langchain.agents.middleware

ModelFallbackMiddleware

Falls back to a different model (or provider) after repeated failures. Chains cleanly with a Groq→Anthropic fallback since both are already wired in provider.py.

langchain.agents.middleware

ToolRetryMiddleware

Retries a failing tool call with backoff, scoped to specific tools if needed. Sentinel's tools already return strings instead of raising, so this is a second line of defense, not the primary fix.

langchain.agents.middleware

ModelCallLimitMiddleware

Caps model calls per run and per thread, ending cleanly instead of looping forever. Cheap insurance against a confused agent burning the free-tier daily quota.

langchain.agents.middleware

SummarizationMiddleware

Compresses older messages once the conversation crosses a token threshold, so long sessions don't blow the context window.

langchain.agents.middleware

ContextEditingMiddleware

Clears old tool-call outputs (e.g. long leaderboards) once total tokens cross a trigger, keeping only the most recent N results.

langchain.agents.middleware

HumanInTheLoopMiddleware

The built-in version of what confirm()/interrupt() already hand-rolls per guarded tool. Not a priority to swap — the existing rail is tested and matches the id-addressed multi-confirmation contract already in the API.

langchain.agents.middleware

PIIMiddleware

Redacts emails, card numbers, IPs in input/output. Low priority for a single-user learning backend, but the right answer if this ever takes real user data.

langchain.agents.middleware

Full catalog is larger — ToolCallLimitMiddleware, LLMToolSelectorMiddleware, TodoListMiddleware and others exist too. These eight are the ones that map to a category Sentinel actually has exposure to.

Multiple middleware, one call each

Wrap-style hooks (wrap_model_call, wrap_tool_call) nest: the first middleware in the list is outermost. Before/after hooks run in registration order going in, reverse order coming out. Order matters when stacking retry + fallback + limits — retries should sit inside the call-limit counter, not outside it, or a retried call won't count against the budget meant to stop it. 

# conceptual order for Sentinel's model-call stack
middleware=[
    ModelCallLimitMiddleware(thread_limit=40, run_limit=15),  # outermost: hard stop
    ModelFallbackMiddleware("claude-haiku-4-5"),                # escalate provider
    ModelRetryMiddleware(max_retries=2, initial_delay=1.0),    # innermost: retry same model first
]
04 / pattern

The corrective-feedback pattern

Middleware handles the mechanics of retrying; it doesn't by itself make a retry useful. A weak model retrying the identical malformed call just fails the identical way. The pattern that actually helps, converging on the same shape across every tool-use reliability study found:

  • Catch, don't crash. The failure becomes a message back into the conversation — not a raised exception the graph has no path around.
  • Name the fix, not just the failure. "Your save_config call is missing framing, failure_threshold, reporting_cadence, success_metric — ask the user for these before calling it again" recovers a turn. "Error: tool_use_failed" doesn't — the model reads "retry" as "call the exact same thing again" unless told otherwise.
  • Cap it. Every retry adds the previous failed attempt to context. A model that fails three times in a row is now reasoning with three failures polluting its own history — cap attempts low (2–3) and fall back to a plain apologetic message past the cap, not an infinite loop.

Applied to the incident: a wrap_model_call middleware that catches the provider's BadRequestError, and instead of retrying the identical request, injects a ToolMessage-shaped correction naming the missing fields before letting the model try again. This is the shape of the eventual Phase-1 fix — not designed yet, just scoped here.

05 / measurement

Observability & evals

Right now the only record of what the agent did is whatever scrolled past in the SSE stream. Three practices, roughly in the order teams actually adopt them:

  • Trace first. LangSmith (or any OpenTelemetry-based tracer) attaches to create_agent with no architecture change — every model call, tool call, and intermediate step becomes inspectable after the fact, instead of inferred from logs. This alone would have made the incident's root cause visible in one click instead of a debugging session.
  • Then eval. Code-based checks (did compare actually re-evaluate on a common rul_cap? did the registry end with the right model active?) catch objective regressions cheaply. LLM-as-judge checks (did the agent's prose explanation actually make sense to a non-expert?) catch quality drift that no assertion can. DeepEval is pytest-native — these could sit next to the existing offline agent tests rather than a separate harness.
  • Close the loop. The highest-leverage version of this: every production failure (like the incident that started this note) becomes a fixture in the eval suite, so a regression can't silently reappear.
06 / scoping

What this means for Sentinel

Per the phased approach: model-call reliability is the natural Phase 1, since it's the category with a live incident, the highest likelihood on the current model, and a direct LangChain-native fix. Rough shape, to be turned into an actual spec next:

  • ModelRetryMiddleware + a corrective-feedback wrap_model_call for the provider-rejection class of failure.
  • A look at whether ChatGroq exposes a strict/constrained-decoding flag through bind_tools, to prevent the malformed call at the source rather than only recovering from it.
  • ModelCallLimitMiddleware as cheap insurance, low effort to add alongside Phase 1.
  • Everything else in the map — context management, observability, evals, PII — deferred to its own later spec, not because it doesn't matter, but because none of it has an incident behind it yet.

Also worth a standalone decision, separate from harness work: the llama-3.3-70b-versatile deprecation clock above.
